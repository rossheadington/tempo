<objective>
Wire the three Wave-1 deliverables (races.py result+completed, heat.py parser+rollup, race_link.py) into the actual analyses. After this plan, `tempo analyze recovery` surfaces heat adaptation status (with the A4 lapsed-nudge override), and `tempo analyze race-readiness` surfaces auto-linked race ↔ activity status. The CLI + daily-sync callers are updated to pass `heat_path` through.

**Scope boundary:** This plan does NOT remove `plan_path` from any signature or call site. Plan E (Wave 3) owns the entire plan.md retirement. Keeping the plan/heat changes separate avoids merge conflicts and lets Plan E focus purely on deletion.

**Open-question override (A4) — this plan implements the user's chosen rule:**
- heat.md missing OR present-but-empty → section omitted entirely.
- heat.md present with sessions in any of the 7/14/28-day windows → render the full rollup line.
- heat.md present with sessions in history but NONE in the last 28 days → render a one-line "heat protocol lapsed — last session N days ago" nudge (the section header + one line, NO rollup numbers).

This three-state degradation is the override of RESEARCH.md A4's "always omit on stale" default.

Purpose: Make the Wave-1 capabilities user-visible. After Plan D, the user can actually see heat adaptation in their recovery report and race-link status in race-readiness.

Output:
- recovery.py extended with HeatRollup field on RecoveryAssessment + heat-section renderer (incl. lapsed-nudge case)
- runner.py: generate_recovery + generate_all gain heat_path param; generate_race_readiness calls race_link
- report.py: render_race_readiness gains race_links param and surfaces link status per race
- CLI (tempo/cli.py) + daily-sync (tempo/sync/daily.py) call sites updated to pass settings.heat_path
- Tests in test_analysis_reports.py, test_analyze_cli.py, test_recovery.py extended for heat + race-link surfacing (and the 3 degradation states)
