    3. **`## Format`** — break out into three numbered/bulleted subsections:
       - **Session header** (`## YYYY-MM-DD [HH:MM] [— Name]`): document the date requirement, optional time, optional name after `—` or ` - ` separator.
       - **Metadata lines**: `rest: M:SS` (converted to `rest_s` in seconds; malformed → None), `notes: <free text>` (verbatim), any other `key: value` line silently ignored.
       - **Exercise bullets** (` - <Name> [(Equipment)] [[GROUP]]: <set>, <set>, ...`): document the name + optional equipment annotation + optional superset group label, then enumerate the three set flavours with examples:
         - Weighted: `WxR` (e.g. `55x8` → 55 kg × 8 reps; lowercase `x`, uppercase `X`, unicode `×` all accepted; whitespace around the separator OK)
         - Timed hold: `M:SS` (e.g. `1:00` → 60 s hold)
         - Bodyweight reps: bare integer (e.g. `15` → 15 unloaded reps)
       - Show the worked Tuesday session as a fenced code block.

    4. **`## Supersets`** — short section explaining the `[A]` / `[B]` convention: two exercises in the same session sharing the same group letter are understood to be supersetted. The parser stores the label verbatim on `StrengthExercise.superset_group`; the renderer / downstream consumers interpret pairings — the parser does NOT enforce that a group letter appears on ≥ 2 exercises.

    5. **`## Lenient parsing`** — the failure-mode contract:
       - Missing file → `StrengthContext(present=False)`, analyses still run, the recovery report omits the `## Strength & conditioning` section.
       - Malformed sets within a valid exercise bullet → silently skipped; the exercise still keeps the sets that did parse.
       - Exercise bullet with zero parseable sets → exercise still kept (sets=()); the rollup contributes 0 to tonnage and 1 to count.
       - Session header with an unparseable date → the entire session block is dropped (the next valid `## YYYY-MM-DD` header re-anchors parsing).
       - Unknown metadata keys → silently ignored.
       - The parser NEVER raises.

    6. **`## Recovery report integration`** — short section: parsed sessions surface in the recovery report as a `## Strength & conditioning` section using a 3-state degradation rule:
       - Absent (file missing or no sessions ever) → section omitted.
       - Lapsed (sessions in history but none in the last 28 days) → one-line nudge: `_S&C protocol lapsed — last session N days ago._`
       - Active (any session in the last 28 days) → full rollup line: `last 7 days: N sessions / X kg tonnage · last 14 days: N sessions / X kg · last 28 days: N sessions / X kg · last session: Name (M days ago)`
       - Tonnage = sum of `weight_kg × reps` over weighted sets only; timed holds and bodyweight reps contribute 0 to tonnage but their parent sessions DO count toward `*_count`.
       - Tonnage formatting: `9,835 kg` for <10t, `12.4 t` for ≥10t.

    7. **`## What's NOT done in this layer`** (optional, ~5 lines) — list the deferred features per CONTEXT lines 252-263:
       - Structured DB tables (`strength_session` / `strength_exercise` / `strength_set`) — deferred.
       - Strong-app CSV importer — deferred.
       - `runos strength add` CLI mirror of `runos journal add` — deferred.
       - Separate tonnage-trend report — deferred.
       - 1RM estimation / PR tracking — out of scope (Strong does this).
       - Strava `WeightTraining` activity ↔ session auto-link — deferred.

    Total document length: ~80-150 lines. Tone: factual + matter-of-fact, mirrors `docs/JOURNALING.md`. No marketing copy; no hyperbole.
  </behavior>
