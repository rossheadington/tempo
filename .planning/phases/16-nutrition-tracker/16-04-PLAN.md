  <behavior>
    `docs/NUTRITION.md` must include the following sections in order. Total length: ~150-260 lines (longer than `docs/WEIGHT.md` because TWO format sections plus the standalone-report + recovery-integration sections):

    1. **`# Nutrition log (`food.md`)`** — title.

    2. **Intro paragraph** — 4-5 sentences: what the file is, who maintains it (the owner, by hand or via agent-append), where it lives (`<content_root>/food.md`, redirectable via `TEMPO_CONTENT_DIR`), why a markdown layer (the Layer-1 surface must prove itself before a structured table is added — out-of-scope per CONTEXT line 25-27; the MyFitnessPal CSV importer is reclassified to v2 NUTR-CSV-01), and the equivalence guarantee (BOTH formats produce identical semantic output).

    3. **`## Two interchangeable formats`** — short overview (4-6 lines): the file accepts BOTH Format A (inline, one entry per bullet) AND Format B (block-per-meal, header + nested bullets) freely intermixed. Choice is purely ergonomic — inline for quick single-item agent-append; block for multi-item meals. Equivalence: a meal parsed from inline produces identical `FoodEntry` records (modulo `source_format` + `source_line`) to the same meal parsed from a block.

    4. **`## Format A — Inline`** — full grammar + a 4-entry example block (use the CONTEXT lines 56-59 example verbatim).
       - Entry shape: `- YYYY-MM-DD <meal>: <food> | p:<g> c:<g> f:<g> cal:<n>`
       - Date: required, ISO format
       - `<meal>`: meal name; lowercased + stripped on parse; free-form (`breakfast`, `lunch`, `dinner`, `snack`, `pre-run`, `post-run`, `4am-snack` all work)
       - `<food>`: opaque verbatim string (`80g rolled oats`, `1 banana`) — the parser does NOT split out quantity
       - The `|` pipe separates the food label from the macros
       - Macros: `p:<g> c:<g> f:<g> cal:<n>` — required, see `## Macro key grammar` below for full spec

    5. **`## Format B — Block-per-meal`** — full grammar + a 2-block example (use the CONTEXT lines 64-73 example verbatim).
       - Header line: `## YYYY-MM-DD <meal>` — required date + optional meal_name (lowercased on parse)
       - Nested bullets: `- <food>: <macros>` — `<food>` is the opaque label; macros use the same grammar as Format A but no leading `<date> <meal>:` prefix
       - A new `##` header (or end of file) closes the previous block
       - Blank lines within a block are tolerated
       - If the header date is malformed → the WHOLE block (header + all nested bullets) is skipped and recorded as malformed
       - Best for multi-item meals (lunch with chicken + rice + greens + dressing as separate macro-attributed components)

    6. **`## Macro key grammar`** — shared by both formats:
       - 4 required keys: `p:<float>` (protein g), `c:<float>` (carbs g), `f:<float>` (fat g), `cal:<int>` (kcal; rounding-tolerant — `cal:303.4` → 303)
       - Key order is FREE — `cal:303 p:13 c:54 f:6` parses identically to `p:13 c:54 f:6 cal:303`
       - Keys are case-insensitive — `P:13 C:54 F:6 CAL:303` works; stored lowercased
       - Whitespace around `:` and between keys is tolerated
       - Unknown keys (`fibre:5`, `sodium:230`) are silently ignored — the parser is forgiving so future micronutrient extensions don't break existing logs
       - Missing ANY of p/c/f/cal → entry skipped (line captured in `malformed_lines`)
       - Float-parse failure on p/c/f → skipped; int-parse with rounding tolerance on `cal:`

    7. **`## Lenient parsing`** — the failure-mode contract (mirror `docs/WEIGHT.md`):
       - Missing file → `FoodContext(present=False, entries=(), blocks=(), path=None, malformed_lines=())`. Analyses still run; both the standalone report (`tempo analyze nutrition`) and the recovery report's `## Nutrition` section degrade gracefully.
       - Malformed lines (bad date, missing macros, etc.) → skipped; the 1-indexed line number lands in `FoodContext.malformed_lines`. The parser NEVER raises.
       - Block headers with bad dates → the whole block is skipped; each affected line is recorded as malformed.
       - Comments (lines starting with `#`) — single-hash headers (`# Food log`) are silently ignored; only `##` (double-hash) is parsed as a Format B header.
       - Blank lines and non-bullet prose lines are silently ignored.
       - The parser does NOT log entry contents (privacy — only line numbers are surfaced).

    8. **`## Latest-wins on duplicates`** — short section: if the same `(date, meal_name, food_label)` triple appears twice in the file, the LAST occurrence in file order wins. Different `food_label`s under the same `(date, meal_name)` are kept as separate entries (a multi-item meal). To correct an earlier entry, append a new line with the same triple — no need to edit existing lines.

    9. **`## Daily aggregation`** — `daily_nutrition(entries, day)` sums all P/C/F (floats) + kcal (int) for one date. Macro percentages computed from KCAL-SHARE AFTER summation:
       - `macro_pct_protein = (protein_g * 4) / kcal * 100`
       - `macro_pct_carbs = (carbs_g * 4) / kcal * 100`
       - `macro_pct_fat = (fat_g * 9) / kcal * 100`
       - When `kcal == 0` (degenerate sum), all three percentages return `0.0` rather than crashing.

    10. **`## Rollup semantics`** — `nutrition_rollup(entries, today, target_kcal=None)` math:
        - All averages use left-open right-closed windows `(today - N, today]`. A same-day log counts; `today - N` itself is EXCLUDED.
        - `latest_day` = the most recent date with entries; `days_since_last = (today - latest_day.date).days`
        - `avg_7d`: the per-day mean across DAYS WITH ENTRIES in the 7-day window (a day with zero entries is NOT counted in the denominator). Returns a `DailyNutrition` with averaged grams + kcal + recomputed macro percentages.
        - `days_logged_7d`: count of distinct dates with ≥ 1 entry in the 7-day window — surfaced for transparency.
        - `avg_28d_kcal`: scalar 28-day trailing mean kcal (int) — deeper 28d trend deferred to v2.
        - `target_kcal`: optional, sourced from `Settings.target_kcal_default` (which reads `TEMPO_TARGET_KCAL`); when unset, both `target_kcal` and `deficit_surplus_7d` are `None` (silent).
        - `deficit_surplus_7d`: `avg_7d.kcal - target_kcal` (positive = surplus, negative = deficit). None when either is None.

    11. **`## Standalone report (`tempo analyze nutrition`)`** — short section: `tempo analyze nutrition` writes `reports/<YYYY-MM-DD>-nutrition.md` containing five sections: `## Today's totals`, `## Per-meal breakdown` (omitted when today has zero entries), `## 7-day rolling average`, `## 28-day kcal mean`, `## Goal` (omitted when `target_kcal_default` is unset). The top-level `tempo analyze` aggregator runs this alongside the existing four reports.

    12. **`## Recovery report integration`** — short section: parsed entries surface in the recovery report as a `## Nutrition` section using a 3-state degradation rule:
        - **Absent** (file missing OR zero entries ever) → section omitted entirely.
        - **Stale** (`days_since_last > 3`) → one-line nudge: `_Last food entry N days ago — log today's meals to keep the rollup live._`
          (Threshold is 3 days, not 14 like weight — food is logged daily.)
        - **Current** (`days_since_last <= 3`) → 7-day-trailing line: `7d avg P:{p}g · C:{c}g · F:{f}g · cal:{kcal} ({D} days logged of 7)`. When `target_kcal` is set, append a second line: `Target {target} kcal/day · 7d Δ {±X} kcal/day` (Unicode `+` / `−` / `±`).
        - Section placement: AFTER `## Weight`, so the recovery report's non-running-context cluster reads `Heat → Strength → Weight → Nutrition`.

    13. **`## Optional goal tracking (`TEMPO_TARGET_KCAL`)`** — short section: set `TEMPO_TARGET_KCAL=2200` in `.env` to enable goal tracking. The `## Goal` section of the nutrition report and the goal-suffix line in the recovery `## Nutrition` section then surface a signed delta. When unset (the default), both surfaces silently omit the goal display.

    14. **`## Agent-append guidance`** — short section: appending a new inline entry is a single-line operation. Write `- YYYY-MM-DD <meal>: <food> | p:<g> c:<g> f:<g> cal:<n>\n` to the end of the file — `cat >> food.md` works. Appending a multi-item meal in Format B is multi-line but still append-only (write the `##` header + the nested bullets all at once). To correct an earlier entry, append a new entry with the same `(date, meal_name, food_label)` triple (latest-wins resolves it). The agent SHOULD NOT modify existing lines.

        - MyFitnessPal CSV importer (`tempo food import <csv>`) — reclassified to v2 `NUTR-CSV-01`. The chosen markdown format is deliberately importer-friendly so the CSV importer becomes mechanical.
        - Structured DB tables (`food_entry`, `meal_block`) — derived from markdown source, rederivable. Layer 2.
        - `tempo food add` CLI command (symmetric with `tempo journal add`) — manual edit or agent-append for now.
        - Per-meal-type rollups (avg breakfast kcal across the week) — Layer 2 once the daily rollup proves itself.
        - Micronutrients (fibre, sugar, sodium, vitamins) — macros + calories only.
        - Food-database lookups / autocomplete — user/agent computes macros at write time.
        - Unit conversion for food quantities — quantity is an opaque string; macros are the source of truth.
