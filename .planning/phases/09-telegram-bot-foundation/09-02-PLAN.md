PTB v22 Update construction for tests:
The research file documents the allowlist behaviour at the filter level: `filters.Chat(chat_id=N)` returns False for any other chat id. The clean way to assert this in tests, without spinning up real Telegram, is to build an `Update` by constructing the dataclasses directly, then call `handler.check_update(update)`.

Use `from telegram import Update, Chat, Message, User` and `from telegram.ext import filters` to build private-chat Update fixtures. `User(id=999, first_name="Tester", is_bot=False)`, `Chat(id=chat_id, type="private")`, `Message(message_id=1, date=now(UTC), chat=chat, from_user=user, text="/start")`, `Update(update_id=1, message=msg)`. PTB supports constructing these without a Bot connection.

For the owner case, `handler.check_update(update)` returns truthy and you can `await start_handler(update, fake_context)` where fake_context has `application.bot_data == {"owner_chat_id": OWNER_ID}` and `update.message.reply_text` is an AsyncMock. For the non-owner case, `handler.check_update(update)` MUST be falsy — proving the filter alone gates the dispatcher (no handler invocation, no log spam, no reply).

A `CommandHandler` exposes `.filters` and `.check_update(update)` — preferred over reaching into PTB internals.

Async test idiom (asyncio):
PTB depends on asyncio. Tempo's existing async CLI tests (tests/test_garmin_cli.py) handle async via `asyncio.run(...)` inline — mirror that to avoid adding `pytest-asyncio` as a new dev-dep.
