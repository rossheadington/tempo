# Plan 18-02: Coros wellness transform

**Wave:** 18-02 (parallel with 18-03 and 18-05 after 18-01 ships)
**Status:** Ready

## Scope

Project Coros raw wellness payloads (sleep / hrv / heart_rate) onto the existing `wellness_day` table. Implement the **per-(day, metric) priority resolver**: Coros wins where it has a value, Garmin fills the gap.

## Files created

- `runos/transforms/coros_wellness.py` — pure no-network transform. Reads `raw_response WHERE source='coros' AND endpoint IN ('sleep','hrv','heart_rate')`, projects each payload onto the metric columns of `wellness_day`. Writes via UPSERT keyed on `(day)`. **Only updates columns where the Coros payload has a non-NULL value** — preserves any value Garmin's transform already wrote for the same day (this IS the priority resolver, executed via SQL `COALESCE(?, existing_col)`).
- `tests/test_coros_wellness_transform.py` — exactly the 6 tests in CONTEXT.md § Test scope under `tests/test_coros_wellness_transform.py`.

## Files modified

- `runos/transforms/runner.py` — sequence the new transform AFTER the Garmin wellness transform. Order: `transform_garmin_wellness` → `transform_coros_wellness` → ... (priority resolver is implicit: Coros writes second, only updating columns where it has values, so Garmin's NULL-fills survive AND Coros's non-NULL overwrites land).

## Files NOT touched

- `runos/transforms/wellness.py` (Garmin's transform) — do NOT modify. The priority resolver is implemented entirely in the Coros transform via per-column COALESCE-on-update.
- `runos/migrations/*` — `wellness_day` schema is unchanged.
- All other modules from other waves.

## Implementation notes

- **The resolver mechanism**: when Coros has a value, UPDATE that column. When Coros has NULL, leave the column alone (so Garmin's prior write survives). Encode this as a per-column conditional in the UPDATE statement:
  ```sql
  UPDATE wellness_day SET
      resting_hr        = COALESCE(?, resting_hr),
      hrv_last_night    = COALESCE(?, hrv_last_night),
      hrv_status        = COALESCE(?, hrv_status),
      sleep_score       = COALESCE(?, sleep_score),
      sleep_seconds     = COALESCE(?, sleep_seconds),
      deep_s            = COALESCE(?, deep_s),
      rem_s             = COALESCE(?, rem_s),
      light_s           = COALESCE(?, light_s),
      awake_s           = COALESCE(?, awake_s),
      body_battery_high = COALESCE(?, body_battery_high),
      body_battery_low  = COALESCE(?, body_battery_low),
      stress_avg        = COALESCE(?, stress_avg),
      steps             = COALESCE(?, steps)
  WHERE day = ?
  ```
  When no row exists for the day yet, INSERT a fresh row from the Coros payload alone (Coros-only day; Garmin transform either didn't run or had no data).
- **Coalesce semantics**: this works because Coros's value, when non-NULL, overrides whatever Garmin wrote; when Coros's value is NULL, the existing column value (which Garmin wrote, or which is NULL if neither has anything) is preserved.
- **Tests** must cover the dual-source matrix: Coros-only day, Garmin-only day, both-sources day with Coros winning per metric, both-sources day where Coros only has SOME metrics and Garmin fills the rest.
- **Parsing helpers**: Coros payload shapes differ from Garmin. Reference cygnusb/coros-mcp's parser logic OR write fresh against the raw payload shape captured in 18-01.
- **Bucketing**: use the LOCAL calendar date Coros assigns to each payload (their `day` field is `YYYYMMDD`; convert to ISO `YYYY-MM-DD` for the `wellness_day.day` PK). No timezone gymnastics; Coros already buckets on the wake-up day for sleep/HRV, same convention as Garmin (see `docs/DATE_BUCKETING.md`).
- **Idempotence**: re-running the transform on the same raw produces the same `wellness_day` rows. Required for `runos rederive`.
- **Defensive**: malformed Coros payload (unexpected JSON shape) → log a warning, skip the day, continue. Never raise; mirror the Garmin transform's defensive parsing.
