<objective>
Wire Plan 11-01's session store and Plan 11-02's agent SDK into the actual Telegram message pipeline. After this plan: a voice memo or text message from the owner produces a Claude Code agent turn whose final reply is sent back as HTML, the conversation persists for 4 hours, `/new` resets the session, tool-call noise stays out of the chat, the typing indicator hides the latency, and per-turn token usage is logged.

Purpose: This is the integration layer where VOICE-07/08/09/10/13 all become observable from a Telegram chat. The agent module and session store are pure-Python contracts; this plan composes them with `python-telegram-bot` handlers and ensures the startup-time `claude` CLI check fails loudly. Plan 12 will wrap this in a launchd plist + top-level error boundary; this plan deliberately stops at "let exceptions propagate to PTB's default handler" per 11-CONTEXT.md `<decisions>` "error handling (LIGHT in Phase 11)".

Output:
- `tempo/bot/handlers.py` — new `text_handler`, new `new_command_handler`, REWORKED `voice_handler` (post-transcription path replaced), and a private `_keep_typing` coroutine.
- `tempo/bot/app.py` — `build_application` registers `text_handler` (filter: TEXT & ~COMMAND & owner) and `CommandHandler("new", new_command_handler, filters=owner)`; also stashes the Tempo project root path + a sqlite3 connection factory in `application.bot_data`. `run()` (or a new helper) verifies `shutil.which("claude")` and exits with a clear error if missing.
- `tempo/bot/__init__.py` — re-export the two new handlers.
- `tests/test_bot_handlers.py` — extend with ~5 new tests for the new flows; assert the Phase 10 raw-transcript-echo behaviour is GONE.
- `tests/test_bot_app.py` — add tests for the new handler registration + the `claude` CLI startup check.
- `docs/TELEGRAM_BOT.md` + `README.md` — short "Phase 11 agent loop" notes (one paragraph each) explaining the new behaviour.
</objective>
