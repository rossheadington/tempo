    1. **`# Weight log (`weight.md`)`** — title.

    2. **Intro paragraph** — 3-4 sentences: what the file is, who maintains it (the owner, by hand; the agent may append entries when asked), where it lives (content dir, default `<content_root>/weight.md`, redirectable via `RUNOS_CONTENT_DIR`), why a markdown layer rather than a structured table (the format must prove itself in real use before a DB schema is added — out-of-scope per CONTEXT lines 24-25).

    3. **`## Format`** — break out into bullets matching the LOCKED entry grammar (CONTEXT lines 56-66):
       - Entry shape: `- <YYYY-MM-DD>: <weight> [<unit>] [| notes: <free text>]`
       - Date: required, ISO format, malformed → line skipped
       - Weight: required, parseable as float, malformed → line skipped
       - Unit: optional, defaults to `kg`; `kg` / `lb` / `lbs` accepted; `lbs` normalised to `lb` on the parsed entry
       - Notes: optional; everything after the FIRST `| notes:` is verbatim (subsequent `|` pipes stay in the notes)
       - Header lines (any line starting with `#`) and blank lines are silently ignored
       - Latest-wins on duplicate dates: a later same-date line overwrites an earlier one (intentional — append-only corrections work)
       - Show the 4-entry CONTEXT example (lines 49-54) as a fenced code block.

    4. **`## Lenient parsing`** — the failure-mode contract (mirror `docs/STRENGTH.md`):
       - Missing file → `WeightContext(present=False, entries=(), path=None, malformed_lines=())`. Analyses still run; the recovery report omits the `## Weight` section.
       - Malformed lines (bad date, bad float, unknown unit, out-of-range weight) → skipped; the 1-indexed line number lands in `WeightContext.malformed_lines`. The parser NEVER raises.
       - Out-of-range sanity check: weights must satisfy `20 < kg_equivalent < 500` to survive. This catches typos like `7.24 kg` (decimal slip), `724 kg` (unit slip), `1600 lb` (≈ 726 kg, also too high).
       - Unknown bullet formats (anything that isn't `- YYYY-MM-DD: <float> ...`) → recorded as malformed.
       - Non-bullet, non-header, non-blank lines (prose) → silently ignored (NOT recorded).
       - The parser does NOT log weight values (privacy — only line numbers are surfaced).

    5. **`## Rollup semantics`** — document the `weight_rollup(entries, today)` math:
       - All averages use left-open-right-closed windows: `(today - N, today]`. A same-day weigh-in counts for every window. `today - N` itself is EXCLUDED.
       - `avg_7d`: mean of entries with `today - 7 < e.date <= today`, kg-normalised.
       - `avg_28d`: same with `today - 28 < e.date <= today`.
       - `ewma_trend`: EWMA with `alpha = 0.1` iterated forward through ALL entries up to `today`, in date order. Seeded from the FIRST entry's kg-normalised weight. Half-life is ~7 entries (so ~7 days if logged daily, ~14 days if every other day) — slow trend by design.
       - `delta_vs_28d`: `latest_kg - avg_28d`, in kg.
       - `days_since_last`: `(today - latest_entry.date).days` (≥ 0).
       - **Unit normalisation**: every weight is converted to kg before averaging (`lb * 0.453592`). The rollup's numeric fields are all kg. The `latest_entry` preserves the original unit. `unit_mixed` is True iff both `kg` and `lb` appear in the entries; the flag is purely informational — the rollup still computes cleanly.

    6. **`## Recovery report integration`** — short section: parsed entries surface in the recovery report as a `## Weight` section using a 3-state degradation rule:
       - **Absent** (file missing OR zero entries ever) → section omitted entirely.
       - **Stale** (`days_since_last > 14`) → one-line nudge: `_Last weigh-in N days ago — log a current reading to keep the rollup live._`
       - **Current** (`days_since_last <= 14`) → full rollup line: `{latest} kg today · 7d avg {a7} kg · 28d avg {a28} kg · trend {ewma} kg · {±X.X} kg vs 28d baseline`. When `unit_mixed=True`, a trailing `_(mixed kg/lb in log — normalised to kg)_` caveat is appended.
       - Section placement: AFTER `## Strength & conditioning`, so the report's non-running-context cluster reads `Heat → Strength → Weight`.

    7. **`## Agent-append guidance`** — short section: appending a new entry is a single-line operation. Write `- YYYY-MM-DD: 72.4 kg | notes: ...\n` to the end of the file. No parse-then-rewrite needed; a `cat >> weight.md` from a shell works. To correct an earlier entry, append a new line with the same date (latest-wins resolves it). The agent SHOULD NOT modify existing lines.

    8. **`## What's NOT in this layer`** — short list of deferred features per CONTEXT lines 207-215:
       - Structured DB tables (`weight_entry`) — deferred until the markdown layer proves useful.
       - `runos weight add --kg 72.4` CLI — deferred (symmetric with `runos journal add`, useful if the agent wants a non-markdown path).
       - Body composition (body-fat %, lean mass) — out of scope.
       - Withings / Fitbit / Garmin auto-import — separate phase if it ever happens.
       - Standalone `runos analyze weight` trend report — deferred (the recovery section is enough surface).
       - Goal tracking (target weight + ETA from trend) — deferred.

    Total document length: ~80-160 lines. Tone: factual, matter-of-fact, mirrors `docs/STRENGTH.md`. No marketing copy.
