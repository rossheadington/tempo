      - `async def telegram_error_handler(update: object, context: ContextTypes.DEFAULT_TYPE) -> None`:
        1. Resolve the exception: `err = context.error`.
        2. Resolve chat / update metadata:
           - `chat_id: int | None = update.effective_chat.id if isinstance(update, Update) and update.effective_chat is not None else None`.
           - `update_type: str` = a short label: `"voice"` if `isinstance(update, Update) and update.effective_message and update.effective_message.voice`, else `"text"` if message.text, else `"command"` if message.text and starts with "/", else `"other"`. (Order: voice > command > text > other; voice/text/command checks are best-effort -- on `update` being a non-Update object, return `"non-update"`.)
        3. Log: `logger.exception("Bot handler crashed", extra={"chat_id": chat_id, "update_type": update_type, "exc_type": type(err).__name__})`. Use `logger.exception` so the traceback is included automatically (it inspects `sys.exc_info` -- since we are inside an error handler, the traceback IS the current exception's traceback per PTB docs; if `logger.exception` does not render it from this context, fall back to `logger.error("Bot handler crashed", exc_info=err, extra={...})`).
        4. Send the canonical reply IFF we have a chat_id:
           - `try: await context.bot.send_message(chat_id=chat_id, text=GENERIC_ERROR_REPLY, parse_mode=ParseMode.HTML)`
           - `except Exception as send_err: logger.warning("error_handler: reply failed", extra={"chat_id": chat_id, "send_err_type": type(send_err).__name__})` -- SWALLOW; never re-raise.
        5. Never propagate `err` -- the function returns normally. Per CONTEXT: "If reply itself fails (e.g. the chat is rate-limited), swallow that silently — never re-raise from the error handler."

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Per-handler code → bot worker process | An uncaught exception here previously crashed PTB (and with KeepAlive=true from 12-01 would restart-loop). VOICE-12 mitigates by adding a top-level error boundary. |
| Telegram server → `context.bot.send_message` | Rate limits + transient network errors. The error handler's reply-send itself can fail; mitigated by try/except around the send, swallowing the second failure. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-12-05 | Denial of Service | A single malformed update (e.g. unexpected Telegram payload, claude-agent-sdk timeout) could crash the worker repeatedly under KeepAlive | mitigate | `Application.add_error_handler(telegram_error_handler)` catches every escape; canonical reply + structured log; never re-raises. |
| T-12-06 | Information Disclosure | The error reply could leak internal exception text to the user | mitigate | Reply is a fixed canonical string; the exception text + traceback go ONLY to stderr/logs (`<project>/logs/telegram-bot.stderr.log`). |
| T-12-07 | Repudiation | Rate-limited reply send could silently fail with no log | mitigate | Reply send is wrapped in try/except; failure logs a WARNING with `send_err_type` so the user can see in logs that the chat failed to receive the apology. |
</threat_model>
